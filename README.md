## Что это

Kemano Transfer Cargo (KTC) — транспортный протокол реального времени, работающий поверх UDP. Предназначен для обхода систем глубокой инспекции пакетов (DPI) и средств активного противодействия (ТСПУ), применяемых для цензуры интернета.

Протокол обеспечивает:

- Полную невидимость самого факта использования (нет сигнатур)
- Защиту от принудительного разрыва соединения (RST-инъекции)
- Адаптивную маскировку под легитимные сервисы
- Обход белых списков IP и портов
- Максимально возможную пропускную способность

**CCLH** — *Cipher-locked Cargo, Legend-handshake*. Груз заперт на ключ, рукопожатие подтверждает легенду.

---

## Зачем

Современные DPI/ТСПУ умеют:

- Инжектировать TCP RST для разрыва VPN-туннелей
- Блокировать трафик по сигнатурам протоколов
- Применять белые списки: только определённые IP, порты, SNI
- Анализировать размеры пакетов и тайминги
- Сканировать порты для обнаружения активных сервисов
- Подменять пакеты (spoofing)

> [!NOTE]
> устройства ТСПУ И иное могут анализировать трафик и с момента подключения.
> 

Обычные VPN (WireGuard, OpenVPN, Shadowsocks) детектируются и блокируются. KTC спроектирован так, чтобы его невозможно было отличить от легитимного трафика, даже при знании всех деталей протокола — без корневого ключа.

## Возможности

| Возможность | Описание |
|-------------|----------|
| **Невидимость рукопожатия** | KTC0 и KTCA зашифрованы, нет открытых сигнатур, каждый байт — выход AEAD |
| **Защита от RST** | UDP — нет флагов состояния, нечего инжектить |
| **Слепой сервер** | Не отвечает на невалидные пакеты, порт выглядит закрытым |
| **Привязка к железу** | Аутентификация через TPM / Secure Enclave, HW_MAC_Proof |
| **Jumbo-фреймы** | Payload до 65 535 байт |
| **8 потоков** | Параллельные независимые BBR-контроллеры |
| **Сжатие zstd** | На лету, уровень 3 по умолчанию |
| **Selective ACK** | Батч-подтверждение диапазонов, до 128 блоков |
| **11 профилей маскировки** | YouTube, Zoom, Discord, WebRTC, HTTP/3, Steam, etc. |
| **Обход белых списков** | Cloudflare Spectrum, Google QUIC, ретрансляторы |
| **Динамическая смена профиля** | MASK_TRIGGER без разрыва соединения |

## Как это работает

### Архитектура

```

Клиент KTC                              Сервер KTC
│                                       │
│── KTC0 (156 байт шума) ──────────────▶│  Рукопожатие
│                                       │  (зашифровано на Handshake_Key)
│◀── KTCA (108 байт шума) ──────────────│
│                                       │
│=== CCLH_DATA (потоки 1-7) ===========▶│  Груз
│◀== CCLH_DATA (потоки 1-7) ===========│  (зашифрован на Session_Key)
│                                       │
│── CCLH_HALT (шифр) ─────────────────▶│  Закрытие
│                                       │
X           (тишина)                   X

```

### Три слоя защиты

```

┌─────────────────────────────────────────┐
│              Слой 1: Транспорт           │
│  UDP-датаграммы, зашифрованный шум      │
│  Маскировка под HTTP/3, WebRTC, STUN    │
├─────────────────────────────────────────┤
│              Слой 2: Рукопожатие         │
│  KTC0 / KTCA, зашифрованы на            │
│  Handshake_Key, неотличимы от шума      │
├─────────────────────────────────────────┤
│              Слой 3: Груз (Cargo)        │
│  CCLH_DATA, многопоточный,              │
│  зашифрован на Session_Key              │
└─────────────────────────────────────────┘

```

### Рукопожатие

1. Клиент и сервер заранее (offline) обмениваются `Root_Key` (256 бит)
2. Из `Root_Key` выводятся `Handshake_Key` и `Legend_Key`
3. Клиент отправляет **KTC0** — 156 байт, полностью зашифрованных на `Handshake_Key`
4. Сервер принимает пакет, перебирает `Handshake_Key` всех известных клиентов, пытается расшифровать
5. Если расшифровался — проверяет `Legend_Sig`, `TPM_Quote`, `HW_MAC_Proof`
6. Сервер отправляет **KTCA** — 108 байт, тоже зашифрованных
7. Обе стороны вычисляют `Session_Key` через X25519

**Ключевое свойство:** цензор, перехвативший KTC0, видит 156 байт криптографического шума. Без `Handshake_Key` невозможно:
- Понять, что это KTC
- Отличить от QUIC Initial Packet
- Подделать ответ

### Передача данных

После рукопожатия:

- Данные разбиваются на блоки по 64 KiB
- Блоки распределяются round-robin по 7 потокам данных (потоки 1–7)
- Поток 0 используется для управления: SACK, MASK_TRIGGER, keepalive
- Каждый поток — независимый BBR-контроллер
- Каждый фрейм шифруется ChaCha20-Poly1305 на `Session_Key`
- Nonce детерминирован из `Stream_ID` и `Cargo_Seq` — не передаётся открыто

### Закрытие соединения

- **Честное:** отправляется CCLH_HALT — зашифрованный фрейм с флагом `HALT`
- **По таймауту:** 30 секунд молчания → локальный разрыв
- **Keepalive:** CCLH_ALIVE каждые 15–25 секунд на потоке 0
- ТСПУ не может подделать HALT — не знает `Session_Key`

## Криптография

### Иерархия ключей

```

Root_Key (256 бит, offline, в TPM)
│
├── Handshake_Key = BLAKE3(key=Root_Key, context="KTCv1.9-Handshake")
│
├── Legend_Key    = BLAKE3(key=Root_Key, context="KTCv1.9-Legend")
│
└── (после рукопожатия)
Shared_Secret = X25519(Client_Priv, Server_Pub)
Session_Key   = BLAKE3(key=Shared_Secret, context="KTCv1.9-Session")

```

- **Root_Key** — никогда не передаётся по сети
- **Handshake_Key** — шифрует KTC0 и KTCA
- **Legend_Key** — для Legend_Sig и Legend_Proof
- **Session_Key** — уникален для каждой сессии

### Алгоритмы

| Примитив | Назначение | Стандарт |
|----------|------------|----------|
| ChaCha20-Poly1305 | AEAD-шифрование | RFC 8439 |
| X25519 | Обмен ключами | RFC 7748 |
| BLAKE3 | Хеширование, KDF | Blake3-team |
| HMAC-SHA256 | Подписи легенды | RFC 2104 |
| zstd | Сжатие | RFC 8878 |
| TPM 2.0 | Аппаратная привязка | ISO/IEC 11889 |

### Форматы пакетов

<details>
<summary><b>KTC0 — Cargo Offer (156 байт)</b></summary>

```

Открытая структура:
┌─────────┬──────────────┬────────────────┐
│ Nonce   │  Ciphertext  │  Poly1305_Tag  │
│ 12 байт │  128 байт    │  16 байт       │
└─────────┴──────────────┴────────────────┘

Внутренняя структура (128 байт):
┌────────────┬─────────────────┬────────────┬───────────────┬─────────────┐
│ Timestamp  │ Client_X25519   │ TPM_Quote  │ HW_MAC_Proof  │ Legend_Sig  │
│ 8 байт     │ 32 байта        │ 32 байта   │ 32 байта      │ 24 байта    │
└────────────┴─────────────────┴────────────┴───────────────┴─────────────┘

```

- `Timestamp` — uint64, мс с эпохи UNIX
- `Client_X25519` — эфемерный публичный ключ
- `TPM_Quote` — подписанная TPM-цитата устройства
- `HW_MAC_Proof` — `HMAC-SHA256(Device_Secret, eth0_MAC || "KTC-HW-Bind-v1.9")`
- `Legend_Sig` — `HMAC-SHA256(Legend_Key, Timestamp || Client_X25519 || TPM_Quote || HW_MAC_Proof)`

</details>

<details>
<summary><b>KTCA — Cargo Accept (108 байт)</b></summary>

```

Открытая структура:
┌─────────┬──────────────┬────────────────┐
│ Nonce   │  Ciphertext  │  Poly1305_Tag  │
│ 12 байт │  80 байт     │  16 байт       │
└─────────┴──────────────┴────────────────┘

Внутренняя структура (80 байт):
┌────────────────────┬─────────────────┬───────────────┬──────────────┐
│ Echoed_Timestamp   │ Server_X25519   │ Server_Proof  │ Mask_Profile │
│ 8 байт             │ 32 байта        │ 32 байта      │ 8 байт       │
└────────────────────┴─────────────────┴───────────────┴──────────────┘

```

- `Echoed_Timestamp` — копия из KTC0
- `Server_X25519` — эфемерный публичный ключ сервера
- `Server_Proof` — `HMAC-SHA256(Legend_Key, Server_X25519 || Echoed_Timestamp)`
- `Mask_Profile` — ID профиля маскировки (0–11)

</details>

<details>
<summary><b>Cargo Frame — данные</b></summary>

```

Внутренняя структура (до шифрования):
┌───────────┬───────────┬─────────────┬─────────────┬─────────┐
│ Cargo_Seq │ Stream_ID │ Frame_Flags │ Payload_Len │ Payload │
│ 4 байта   │ 1 байт    │ 1 байт      │ 2 байта     │ N байт  │
└───────────┴───────────┴─────────────┴─────────────┴─────────┘

Frame_Flags (битовая маска):
Бит 0: EXT_LENGTH     — Payload_Len расширяется до uint32
Бит 1: COMPRESSED     — payload сжат zstd
Бит 2: SACK_PRESENT   — содержит Selective ACK
Бит 3: MASK_TRIGGER   — смена профиля маскировки
Бит 4: HALT           — честное закрытие
Бит 5: ALIVE          — keepalive
Биты 6-7: зарезервированы

Шифрование:
ChaCha20-Poly1305(key=Session_Key, nonce=(Stream_ID << 32) | Cargo_Seq)

Открытый фрейм:
┌──────────────┬────────────────┐
│  Ciphertext  │  Poly1305_Tag  │
│  N байт      │  16 байт       │
└──────────────┴────────────────┘

```

Nonce не передаётся — детерминирован. Снаружи сплошной шум.

</details>

<details>
<summary><b>SACK-блок</b></summary>

```

┌────────┬──────────┬──────────────────────┐
│ Count  │ ACK_Seq  │ SACK_Blocks          │
│ 2 байта│ 4 байта  │ (Start+End)*Count    │
└────────┴──────────┴──────────────────────┘

SACK_Block:
┌───────────┬─────────┐
│ Start_Seq │ End_Seq │
│ 4 байта   │ 4 байта │
└───────────┴─────────┘

```

- `Count` — количество блоков (1..128)
- `ACK_Seq` — последний непрерывный Seq
- `SACK_Blocks` — до 128 диапазонов полученных фреймов

</details>

## Высокая производительность

### BBR-контроль потока

Алгоритм на основе измерения доступной полосы пропускания, а не реакции на потери (в отличие от TCP Cubic).

| Фаза | Описание | Длительность |
|------|----------|--------------|
| Startup | Экспоненциальный рост окна | До первой потери |
| Drain | Сброс очереди | 1 RTT |
| ProbeBW | Циклическое зондирование: gain = [1.25, 0.75, 1, 1, 1, 1, 1, 1] | 8 фаз |
| ProbeRTT | Окно падает до 4 пакетов | Каждые 10 сек |

```

BW = max(доставленные_байты / интервал) за последние 8 окон
RTT_min = min(RTT) за последние 10 секунд
CWND = BW * RTT_min * gain

```

### Многопоточность

```

Входной поток:
┌────────┬────────┬────────┬────────┬────────┬────────┬────────┐
│ Блок 0 │ Блок 1 │ Блок 2 │ Блок 3 │ Блок 4 │ Блок 5 │ Блок 6 │ ...
└────────┴────────┴────────┴────────┴────────┴────────┴────────┘
│        │        │        │        │        │        │
▼        ▼        ▼        ▼        ▼        ▼        ▼
Поток 1  Поток 2  Поток 3  Поток 4  Поток 5  Поток 6  Поток 7

```

- 8 потоков (ID 0–7)
- Поток 0 — управляющий
- Потоки 1–7 — данные
- Каждый поток: свой BBR, свой буфер, своя последовательность Seq

### Сжатие

- **Алгоритм:** zstd, уровень 3 по умолчанию
- **Применение:** к Payload каждого фрейма (кроме управляющих)
- **Флаг:** `Frame_Flags.COMPRESSED = 1`

### Selective ACK

- Отправляется каждые 16 полученных фреймов или 10 мс
- До 128 диапазонов подтверждённых фреймов
- Retransmit только отсутствующих

## Адаптивная маскировка

### Профили

| ID | Профиль | Размер (байт) | Тайминг | Порт | Легенда |
|----|---------|---------------|---------|------|---------|
| 0 | RAW | Любой | Любой | Любой | Без маскировки |
| 1 | YouTube Live | 1200–1400 | Всплески 100–200 мс | 443 | QUIC |
| 2 | Zoom Video | 200–400 | 20 мс | 8801 | SRTP |
| 3 | Discord Voice | 80–120 | 20 мс | 443 | WebSocket |
| 4 | WebRTC | 1100–1300 | 10–50 мс | 3478 | STUN/SRTP |
| 5 | HTTPS CDN | 1460 | Bulk | 443 | HTTP/2 |
| 6 | DNS-over-HTTPS | 400–600 | 100–500 мс | 443 | HTTPS |
| 7 | Steam Download | 1400–1500 | Поток | 27015 | Game |
| 8 | STUN/WebRTC | 80–120 | 20 мс | 3478, 19302 | STUN |
| 9 | **HTTP/3 QUIC** | 1200–1400 | QUIC | 443 | HTTP/3 |
| 10 | Cloudflare | Любой | Любой | Любой | QUIC/HTTP/3 |
| 11 | Google QUIC | 1200–1400 | QUIC | 443 | HTTP/3 |

### Механика

1. Сервер выбирает профиль → сообщает в `KTCA.Mask_Profile`
2. Клиент дробит payload на фреймы указанного размера
3. Выдерживает тайминги + джиттер ±10%
4. Недостающие данные → pad случайными байтами
5. Смена профиля через `MASK_TRIGGER` без разрыва соединения

## Обход белых списков

### Стратегии

#### A. Свой IP с легендой
Сервер на IP, где работает реальный HTTPS (nginx + LetsEncrypt). DPI сканирует, видит HTTPS → заносит IP в белый список.

#### B. Cloudflare Spectrum (РЕКОМЕНДУЕМАЯ)

```

Клиент → UDP:443 → Cloudflare Edge (1.1.1.1) → Spectrum → Ваш сервер

```

- IP Cloudflare невозможно заблокировать
- Порт 443 — стандартный HTTPS/HTTP/3
- Spectrum пробрасывает UDP без изменений
- Сервер скрыт за Cloudflare

> [!NOTE]
> Можно использовать Адрес IPv4: 8.8.8.8 8.8.4.4 IPv6: 2001:4860:4860::8888 2001:4860:4860::8844 (принадлежат Google).
> 

#### C. Ретранслятор
Минимальный UDP-проброс на арендованном VPS в разрешённом дата-центре. Не знает ключей.

```python
# relay.py
import socket
sock = socket.socket(socket.AF_INET, socket.SOCK_DGRAM)
sock.bind(("0.0.0.0", 51820))
while True:
    data, addr = sock.recvfrom(65535)
    sock.sendto(data, ("real-server.com", 51820))
```

D. Динамические IP

Пул IP с HTTPS-легендой. Список доставляется клиенту скрыто. Заблокированный IP заменяется.

E. Google QUIC (резерв)

Трафик на IP Google. Блокировка IP Google = разрушение интернета. Не делают.

Безопасность

Модель угроз

Злоумышленник: государственный DPI (ТСПУ), способный:

· Пассивно перехватывать весь трафик
· Активно инжектировать пакеты (RST, ICMP)
· Применять белые списки IP/портов
· Анализировать сигнатуры протоколов
· Сканировать порты
· Не способен: взламывать ChaCha20-Poly1305, X25519, BLAKE3

Устойчивость к атакам

Пассивный перехват Виден шум, неотличимый от QUIC/WebRTC
DPI-сигнатура Нет сигнатур — нет открытых тегов
Инжект TCP RST UDP — нет флагов состояния
Инжект ICMP Игнорируется
Сканирование порта Тишина. Сервер не расшифровывает → DROP
Инжект поддельного KTC0 Не расшифруется без Handshake_Key → DROP
Инжект поддельного KTCA Клиент не расшифрует → DROP
Инжект поддельного Cargo Poly1305 не сойдётся → DROP
Replay-атака Timestamp окно + монотонный Cargo_Seq
Подмена MAC HW_MAC_Proof через TPM
MITM X25519 + Legend_Sig
Корреляция сессий Новые ключи на каждую сессию
DoS-флуд Сервер пробует ChaCha20 → быстро дропает
Перебор легенд Нет оракула — нельзя узнать, существует ли клиент
Компрометация сервера Root_Key уникален для каждого клиента

Быстрый старт

Установка

```bash
# Debian/Ubuntu
sudo apt install ktc

# Arch Linux
sudo pacman -S ktc

# macOS
brew install ktc
```

Генерация ключей

```bash
# На сервере: создать клиента
ktc-server keygen --client-id my-laptop --output client.keys

# Перенести client.keys на клиента (физически, через QR, etc.)
```

Запуск сервера

> [!WARNING]
> Могут Не работать До 2028.
>

```bash
# Минимально
ktc-server --port 51820 --clients-db /etc/ktc/clients.db

# Полная конфигурация
ktc-server --config /etc/ktc/server.yaml
```

server.yaml:

```yaml
listen:
  port: 51820

clients:
  db: /etc/ktc/clients.db

masking:
  default-profile: 0

performance:
  max-streams: 8
  compression: true
```

Запуск клиента

```bash
# Прямое подключение
ktc-client --server 10.0.0.1:51820 --keys client.keys

# Через Cloudflare Spectrum
ktc-client --server 1.1.1.1:443 --profile http3 --keys client.keys

# Полная конфигурация
ktc-client --config client.yaml
```

client.yaml:

```yaml
server:
  ip: 1.1.1.1
  port: 443

profile: http3

keys: client.keys

performance:
  streams: 8
  compression: true
  compression-level: 3

tpm:
  enabled: true
```

Настройка Cloudflare Spectrum

1. Зарегистрируйте домен: example.com
2. Включите проксирование Cloudflare (оранжевое облако)
3. Создайте Spectrum-приложение:
   · Тип: UDP
   · Входящий порт: 443
   · Назначение: ваш-сервер-ip:51820
4. На клиенте: ktc-client --server example.com:443 --profile http3 --keys client.keys
